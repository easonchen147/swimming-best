# 队员完整生日与后台花名册一致性 Requirements

## Overview

这次需求集中在后台 `队员管理` 页的几个高频问题：

- 出生信息目前只能录到年份，无法录到月和日；
- 日期选择器仍然是自定义原生 `type="date"` 组合，样式和全局组件不一致；
- 队员表格在 PC 端列头与内容对不齐，姓名列还重复拼接了队伍名；
- 队员编辑时，把公开状态改成“完全隐藏”后，再切回公开会失效；
- 队员表格操作列里仍保留低价值的“下载”按钮。

这不是一个单点 UI 微调，而是一次围绕“队员后台主工作流”的完整修正。

## Problem Frame

### A. 出生信息粒度不够，且升级必须兼容旧库

当前后台只能维护 `birthYear`。实际运营中，管理员需要录入完整生日，后续无论是年龄判断、成长分析还是更精细的分组，都离不开月日信息。

但系统已经有历史数据，旧库里只有 `birth_year`。这意味着新版本不能粗暴重写旧字段，也不能伪造历史生日。

### B. 日期组件不符合当前全局设计语言

现在的日期控件本质上还是一个定制 Popover 包原生日期输入框。它虽然能用，但视觉、交互和项目现有 shadcn / Radix 组件族不一致，也和你给出的 shadcn Calendar 参考不一致。

### C. 队员表格的信息层级和对齐都不够干净

当前姓名列把队员名和队伍名混在一列里，再加上右侧队伍列又单独展示队伍，信息重复。与此同时，PC 端表头宽度和内容宽度没有收口到一个稳定结构，导致“公开展示”“真实姓名”等区域读起来是歪的。

### D. 队员公开状态的编辑链路存在严重可逆性 bug

“完全隐藏”是一个合法状态，但它不应该变成一次性的死胡同。当前现象说明管理员把队员改成隐藏后，前端或请求载荷没有把后续切回公开所需的状态完整提交，导致数据无法恢复到公开态。

### E. 低价值动作污染了高频表格操作区

表格操作列里当前既有编辑，也有摘要导出和原始下载。你已经明确指出“下载”按钮没什么用，这意味着它在高频工作流里属于噪音，应该移除。

## Users

- 教练 / 管理员：需要高效维护队员基础档案，尤其是生日、公开状态、队伍归属和表格浏览。
- 维护者：需要一个不会破坏历史数据的升级方案，并且让前后端字段语义保持一致。

## Goals

- 后台队员档案支持录入完整生日，前后端都能读写。
- 新版本升级时兼容仅有 `birth_year` 的历史数据库。
- 队员生日控件改为与全局风格一致的 shadcn / Radix Calendar 交互。
- 队员表格在 PC 端列头与内容对齐，姓名列只展示队员身份，不再重复拼队伍名。
- 修复“完全隐藏”后无法切回公开的严重 bug。
- 移除队员表格操作列里的“下载”按钮。

## Non-Goals

- 不在公开页暴露完整生日。
- 不重做整个后台页面的视觉主题。
- 不在本轮引入新的年龄计算规则或新的分组产品能力。
- 不批量回填旧数据的具体月日，避免制造伪精确数据。

## Key Product Decisions

### 1. 完整生日是新的首选字段，但旧年份仍要保留兼容价值

推荐方向：

- 新增可选的完整生日字段，例如 `birthDate`；
- 继续保留兼容性字段 `birthYear`；
- 当管理员保存完整生日时，系统自动同步兼容年份；
- 当历史记录只有年份没有生日时，继续按旧年份工作，不强制补月日。

### 2. 不伪造历史生日

旧库如果只有 `2016`，就仍然是“只知道 2016”。  
不把它自动补成 `2016-01-01` 或任何其他默认日期。

### 3. 队员编辑的公开状态必须是可逆的

推荐将“公开勾选”和“展示姓名模式”视为两个需要一起提交的状态，而不是让某个前端局部状态悄悄覆盖另一个状态。  
“完全隐藏”可以强制落成隐藏态，但管理员后续改回 `展示昵称 / 展示真名` 并重新勾选公开后，系统必须恢复公开。

### 4. 表格按“信息去重 + 结构对齐”原则收口

推荐方向：

- 姓名列只承载队员身份本身；
- 队伍列单独展示队伍；
- 操作列收敛到真正高频动作；
- 表头和内容列宽在桌面端显式约束，避免读起来错位。

### 5. 日期组件遵循你给出的 shadcn Calendar 参考

不继续扩展原生 `type="date"` 的包装层。  
直接采用 shadcn 官方推荐的 `Popover + Calendar` 结构，并接入当前项目已有的视觉语言。

## Recommended Direction

推荐把这轮工作定义为 **“队员后台核心工作流一致性修复”**：

- 数据层：补完整生日字段并兼容历史库；
- 交互层：把生日控件升级为 shadcn Calendar；
- 页面层：修复表格对齐、去掉重复信息、删除低价值动作；
- 正确性层：修复公开状态编辑 bug，并加回归测试。

这个方向的价值最高，因为它一次性解决了数据、UI 和状态正确性的三类问题，而且都集中在同一个高频后台页面里。

## Alternatives Considered

### A. 只把年份输入替换成完整日期，不考虑旧数据兼容

不推荐。  
现有库已经有历史数据，这样会直接把升级风险留给线上。

### B. 用默认月日把旧年份补成完整生日

不推荐。  
这会制造看似精确、实际不真实的数据。

### C. 只修 bug，不顺带处理表格和日期组件样式

不推荐。  
因为你已经明确指出这个页面存在多个可见缺陷，而且都聚焦在同一工作流，拆成多轮只会反复进入同一文件。

## Success Criteria

- 后台创建 / 编辑队员都可以提交完整生日。
- 升级旧数据库后，已有队员记录继续可读可写，旧 `birthYear` 不丢失。
- 出生日期控件的 placeholder 只显示“请选择”。
- 日期控件采用 shadcn Calendar 风格，和全局组件族一致。
- PC 端队员表格列头和内容对齐，姓名列不再拼接队伍名。
- 队员从“完全隐藏”可以稳定切回公开状态。
- 队员表格操作列不再出现“下载”按钮。
