# 游泳成绩管理系统设计

## 背景

本项目面向儿童游泳训练场景，目标是把分散在纸质记录、聊天消息和临时表格里的成绩，收敛成一套可持续维护、可公开展示、可对比分析的成绩管理系统。系统首版只提供管理员登录，由管理员统一维护孩子档案、结构化项目、训练/测试/比赛成绩和阶段目标；公开访客无需登录即可查看被允许公开的成绩展示页。

项目从第一版开始就要兼顾两类访问场景：

- PC 端后台录入与回看
- 手机端公开浏览、快速查看 PB、目标进度和对比结果

## 目标

- 支持多孩子档案与公开身份模式控制
- 支持结构化游泳项目定义，保证 PB、趋势和对比的统计口径稳定
- 同时支持单条快速录入和训练课/测试/比赛上下文录入
- 自动识别 PB、阶段最佳、目标进度和同项目多人对比
- 提供公开成绩页、分享页，并确保桌面端和移动端都可用
- 技术栈固定为 `Go 1.24.1 + Gin + SQLite` 与 `Next.js + React + shadcn/ui + Tailwind CSS`

## 非目标

- 不做对外注册登录
- 不做支付、报名、通知、排课等教培平台功能
- 不做复杂预测模型和 AI 训练建议
- 不做跨项目混合排名

## 关键设计决策

### 1. 使用“记录上下文 + 成绩项”的统一模型

单条录入和批量录入在界面上是两种体验，但底层统一映射为：

- `record_contexts`：记录一次训练、测试、比赛或单条录入的上下文
- `performances`：挂在上下文下面的具体成绩项

这样既能保留“今天这次训练/测试”的语义，又不会让数据模型裂成两套。

### 2. 项目定义采用结构化字段 + 标签补充

核心项目维度必须结构化：泳池长度、项目距离、泳姿、类型。PB 和分析都以结构化项目为准。训练条件、选拔、月测、带脚蹼等灵活信息由标签和备注承载，避免统计口径被自由文本污染。

### 3. 管理端与公开端在一个 Next.js 应用内分区

前端采用一个 Next.js 应用承载两套体验：

- `(admin)`：后台录入与管理
- `(public)`：公开查看、趋势展示、分享导出

这样设计可以共享组件、图表和样式变量，并且更容易保持桌面与移动端的一致体验。

### 4. 管理员账号写配置，但密码必须以哈希保存

虽然账号密码来自配置文件，但密码不以明文保存，而是使用 bcrypt 哈希。后端登录成功后发放 `HttpOnly` 会话 Cookie，避免把管理凭据放到前端或 URL 中。

### 5. SQLite 作为 MVP 存储，启用 WAL 与外键

SQLite 满足单管理员、低部署成本、可本地运行的要求。数据库连接启动时启用：

- `PRAGMA foreign_keys = ON`
- `PRAGMA journal_mode = WAL`

这样可以保证基础一致性与读写体验。

### 6. 趋势展示采用“原始成绩 + PB 包络线 + 平滑趋势线”

首版不做夸张预测，而是展示三层信息：

- 原始成绩点：真实波动
- PB 包络线：最好成绩如何被不断刷新
- 平滑趋势线：最近一段时间的总体方向

这样既诚实，又足够直观。

### 7. 响应式设计是首版硬要求，不是视觉补丁

首版页面必须同时支持：

- 后台在桌面端高效录入，在手机端可浏览与基本维护
- 公开页在手机端优先保证信息层级、图表可读性和分享体验

布局原则：

- 桌面端：信息面板 + 图表 + 表格并置
- 移动端：改为卡片堆叠、抽屉筛选、分段图表与精简表头

## 风险与取舍

- `[风险] 功能过多导致首版拖慢` → 通过“成绩中台 MVP”范围控制，不做教培平台延伸功能
- `[风险] 自由文本项目名导致统计混乱` → 采用结构化项目模型
- `[风险] 分享图服务端渲染成本高` → 首版使用专用分享视图 + 前端导出
- `[风险] 图表在移动端拥挤` → 首版只保留最关键图层与时间范围切换
- `[风险] 管理后台手机端可用性差` → 从页面结构和表单组件层面按移动优先适配

## 交付与迁移

项目是绿地仓库，不涉及老系统迁移。首版交付按以下顺序推进：

1. 完成文档、计划和 OpenSpec 变更
2. 搭建前后端骨架和基础配置
3. 实现后台主数据与成绩录入
4. 实现 PB、趋势、目标和对比 API
5. 实现后台与公开端页面
6. 补齐自动化测试、手工验证、OpenSpec 归档
