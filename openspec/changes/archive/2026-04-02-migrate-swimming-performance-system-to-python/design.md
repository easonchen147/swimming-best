## Context

当前仓库已经存在一个未归档的 `build-swimming-performance-system` 变更，并且代码层面已经实现了部分 Go 后端和部分 Next.js 前端。现状的问题不是“从零实现”，而是“已有半成品与新目标栈冲突”：后端必须改为 Python，同时系统还缺少前端管理与公开体验的收尾，以及孩子 team/group 的完整业务支持。

新的变更必须同时满足三个约束：一是保持产品意图不变，继续提供固定管理员登录保护、后台录入与公开展示；二是尽量保持 API 形状稳定，减少前端重写成本；三是在迁移完成后仓库内不再保留任何 Go 资产或运行入口。

## Goals / Non-Goals

**Goals:**
- 用 `Flask + SQLite` 重建后端，完整承接原有登录、管理、分析和公开 API 能力。
- 用最小但完整的 Python 工具链交付后端：`Python + uv + Poetry + Flask + SQLite`。
- 在泳者档案、录入流程、公开页面和测试中贯通 team/group 能力。
- 补完旧变更未完成的后台页、公开页、响应式行为、文档和验证流程。
- 在验证通过后删除全部 Go 相关资产，并完成双变更归档。

**Non-Goals:**
- 不引入额外账户体系、家长注册、多租户或权限分级。
- 不新增排课、缴费、通知、训练计划等超出原 MVP 的业务域。
- 不引入 ORM、消息队列或独立缓存层等超出当前规模的基础设施。

## Decisions

### Decision: 使用单 Flask 应用承接后台与公开 API

后端改为一个 Flask 应用工厂，统一注册 `/api/admin/*` 与 `/api/public/*` 路由，并保留 `/healthz`。后台鉴权继续采用固定管理员 + Cookie Session 的简单模式，避免在迁移时扩展鉴权复杂度。

备选方案：
- 使用 FastAPI 可获得更强的类型约束，但用户已经明确指定 Flask。
- 拆成多个 Flask blueprint 服务没有当前规模上的必要，只会增加启动与测试复杂度。

### Decision: 继续使用 SQLite，并采用标准库 `sqlite3` + 显式 schema 初始化

数据库继续使用 SQLite，保留简单、可本地运行的特性。数据访问层使用标准库 `sqlite3` 与集中 schema 初始化，不引入 ORM。这样可以降低依赖数量，让迁移聚焦在业务逻辑与 API 行为而不是数据框架切换。

备选方案：
- 使用 SQLAlchemy 会提高抽象一致性，但对当前 MVP 来说属于额外复杂度。
- 使用外部数据库不符合最小完整架构目标。

### Decision: 尽量保持现有前端消费的 API 形状稳定

为了缩小前端修改面，新的 Python 后端将尽量延续当前 Go API 的路径、主要请求体字段和核心响应结构，同时补上 team/group 字段、列表接口和页面需要的额外返回内容。Next.js 重写规则可以继续把 `/api/*` 代理到 Python 后端。

备选方案：
- 全量重设计 API 会让前端返工范围扩大，不适合这次迁移。

### Decision: team/group 作为泳者主档字段，而不是独立复杂实体

当前业务需求只需要 A/B/C 等分组，用于区分孩子和辅助展示，不需要独立团队管理域。因此将 `team/group` 作为泳者档案中的一个字符串字段处理，并在列表、筛选、详情和录入界面中使用。只有当未来出现独立团队管理需求时，再抽象为单独表。

备选方案：
- 建立独立 team 表和外键约束会让当前需求过度设计。

### Decision: 先以测试锁定行为，再迁移实现，再删除 Go

后端会先写 Python 侧测试覆盖登录、泳者管理、成绩录入、分析与公开 API；前端会补充组件和页面测试。待 Python 版本通过验证后，再彻底删除 Go 文件和脚本引用，避免“删太早导致无可对照”的迁移风险。

备选方案：
- 先删除 Go 再重写 Python，会让行为对照与回归验证失去参照。

## Risks / Trade-offs

- `[前后端接口在迁移中出现细微不兼容]` → 以现有前端调用与旧后端测试为基线，优先保持路径和字段稳定。
- `[team/group 展示范围过大导致页面膨胀]` → 仅在区分身份、筛选或公开上下文确有价值的位置展示，不扩展为独立业务中心。
- `[Poetry 与 uv 的组合使用不清晰]` → 以 `pyproject.toml` 为单一依赖声明源，README 中明确区分 `poetry install/run` 与 `uv run/sync` 的推荐用法。
- `[旧 Go 文件残留导致仓库状态不一致]` → 在 Python 后端与验证完成后统一做一次针对 Go 资产的清理，并通过全文搜索核查残留引用。
- `[旧变更与新变更都未归档，规格同步顺序容易混乱]` → 先完成并归档旧变更，再归档新变更，确保主 specs 的基线与增量顺序一致。

## Migration Plan

1. 创建并完善新的 OpenSpec 变更工件，明确迁移与 team/group 的规格范围。
2. 在 `backend/` 下建立 Poetry 驱动的 Flask 项目、测试基座和 SQLite schema。
3. 以测试驱动实现登录、泳者、项目、成绩、目标、分析与公开 API，并加入 team/group 字段。
4. 更新前端 API 类型与页面，补完后台/公开页缺口与 team/group 展示。
5. 运行前后端完整验证，通过后删除全部 Go 代码、脚本和文档引用。
6. 更新 README、运行脚本、OpenSpec 任务状态，执行验证、归档旧变更与新变更，并提交最终 commit。
