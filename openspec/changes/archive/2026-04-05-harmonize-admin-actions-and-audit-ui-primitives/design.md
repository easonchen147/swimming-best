## Context

当前后台头部动作和快速操作区已经共享同一套 `Button` primitive，但页面层的组合方式还不够严格。头部两个按钮使用了不同的 variant 语气与文字色彩，导致视觉上像来自两套系统；快速操作区三个入口虽然都使用按钮，但内部容器结构并不完全同构，因此在左对齐、图标块和箭头对位上仍会显得不平。

同时，用户要求对整个项目的 shadcn 组件使用方式做一次系统审计。现状上，`Button` / `Select` / `Label` / `Badge` 已经完成了关键语义收口；`Card` / `Input` / `Table` 保留了项目级视觉与 motion 定制，但并不属于“语义没切对”的问题。这轮需要把这个审计结论明确下来，而不是继续无差别重写所有 primitives。

## Goals / Non-Goals

**Goals:**
- 让后台头部两个按钮保持统一字重、尺寸和基准布局。
- 让后台首页三个快速操作入口在结构上完全同构，做到严格左对齐。
- 把 shadcn 全局审计结论显式同步进 specs，并在必要处补最小测试覆盖。

**Non-Goals:**
- 不重绘整站配色。
- 不重新设计 public 页或 compare 页。
- 不把所有项目级视觉定制强行改回官方默认样式。

## Decisions

### 1. 后台头部按钮统一为同一轮廓语义
**Decision**: “查看公开页”和“退出登录”都采用同尺寸、同字重、同圆角、同 outline-like 胶囊按钮，仅通过边框和背景的轻语义色差区分。
**Rationale**: 用户指出的是“字体样式和颜色不协调”，本质是按钮层级语义不统一，而不是缺少差异。
**Alternative considered**: 保留危险实底按钮。被拒绝，因为它会持续打断头部动作区的整体节奏。

### 2. 快速操作改为数据驱动统一模板
**Decision**: 把后台首页三个快速入口抽成同一份配置数组，通过单一模板渲染，确保图标容器、文案块、箭头和对齐方式完全一致。
**Rationale**: 这比手写三个相似 JSX 块更可靠，也能避免后续只改一个按钮忘记另外两个。
**Alternative considered**: 继续手写三个按钮，仅微调 className。被拒绝，因为容易再次漂移。

### 3. shadcn 审计结论写入 spec，而不是继续扩大改动
**Decision**: 在 `uiux-modernization` 中明确“官方 primitive 语义优先，项目视觉定制允许存在”，并将本轮审计结论作为约束同步。
**Rationale**: 这样能区分“错误实现”与“有意定制”，避免未来继续把正确的项目风格误判为未对齐。

## Risks / Trade-offs

- [Risk] 两个头部按钮过于接近，降低退出登录的风险提示 → Mitigation: 保留轻度 rose 语义色差，但不再使用强烈实底。
- [Risk] 快速操作模板统一后，某个按钮的特殊信息密度可能被压缩 → Mitigation: 允许副标题文本不同，但版式必须相同。
- [Risk] 审计结论如果写得太宽泛，会让后续执行标准继续模糊 → Mitigation: 在 spec 中明确“根语义必须官方对齐，视觉定制可保留”的边界。
